## Conversation terminology delta

The visible English conversation vocabulary moved from **task** to **chat**
across loading, creation, handoff, archive, pin, rename, parent/fork, side-chat,
and app-control activity labels.

The public kit does not expose these product workflows directly, but its
default and showcase copy follows the current vocabulary:

- `Loading task…` became `Loading chat…`.
- New-chat examples use **chat**, not **task**.
- Plan step terminology remains independent; a plan can still report completed
  tasks supplied by the host.

## Context compaction and optimization delta

The current Renderer preserves three context lifecycle modes with separate
running and completed labels:

| Mode | Running | Completed |
| --- | --- | --- |
| Manual | `Compacting context` | `Context compacted` |
| Automatic | `Context automatically compacting` | `Context automatically compacted` |
| Work | `Optimizing the conversation` | `Optimized the conversation` |

Running labels use the existing loading-text treatment. Completed labels settle
into the compact conversation-summary treatment. Both are transcript items,
not modal application states.

`ThreadContextOptimization` models these states with protocol-neutral mode,
status, label, and icon inputs. The host remains responsible for mapping Codex
events to the public component.

## Handoff and end-resource delta

- Inline handoff activity now distinguishes queued/running, completed, failed,
  and warning states.
- Destination copy distinguishes local, worktree, and a host-provided display
  label.
- The warning state means the handoff needs attention; it is not equivalent to
  failure. The shared activity/status primitives therefore expose a scoped
  `warning` state and semantic warning token without widening unrelated
  command, file-change, search, tool-call, or message status contracts.
- App-generated end resources expose preview, open, and share actions. Existing
  `ResourceCard` action and open slots already cover these host-owned actions,
  so no Codex-specific resource API is required.

## Continuation choice dialog delta

Continuing from a non-latest user message now opens a compact modal with
workspace and worktree choices. Each choice combines a short label, supporting
description, optional icon, disabled state, and host-owned action.

The public implementation models the reusable surface rather than the Codex
fork workflow: `Dialog` owns modal semantics, focus trapping/restoration,
document scroll locking, dismissal, and responsive sizing; `DialogChoice`
provides the descriptive action row. Routing, workspace eligibility, worktree
creation, and submission state remain host-owned.

## Verification boundary

environment. This sample therefore uses read-only packaged metadata, semantic
labels, state branches, and stylesheet continuity as the Codex-side evidence.
The independently implemented geometry, theme behavior, focus behavior, and
responsive layout are verified in the repository's browser showcase and real
Electron BrowserWindow playground. No extracted code, CSS, fonts, images,
icons, sounds, private IPC, or service details are included in the repository.

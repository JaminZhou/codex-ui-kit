## Architecture observations

- The desktop shell is Electron and the primary Renderer is a React
  application built with Vite.
- Renderer styling combines generated utility CSS with semantic CSS custom
  properties for surfaces, text, borders, status, editor colors, and layout.
- The thread is a responsive content column backed by a virtualized turn list.
- A turn can contain user messages, assistant messages, command execution,
  patches, searches, MCP calls, approval review, subagent activity, and errors.
- Related work items are grouped more tightly than unrelated turns.
- Running, completed, failed, and pending states share one activity model rather
  than separate card systems.
- Activity rows can expose expandable details while keeping their summary on a
  compact single line.
- The composer is a separate bottom surface; floating and side panels are
  layered independently from the scrollable thread.

## Independent implementation decisions

- Use a protocol-neutral `AgentActivity` primitive for commands, patches,
  searches, subagents, and arbitrary tools.
- Use native `details` and `summary` semantics for the first expandable activity
  implementation.
- Keep all public class names and tokens under the `codex-ui-` namespace.
- Use system fonts; do not redistribute bundled OpenAI fonts.
- Reproduce measured semantic light and dark roles under the public
  `--codex-ui-` namespace without shipping extracted CSS.
- Keep virtualization and protocol adapters outside the first primitives
  milestone.
- Model the composer as a controlled, protocol-neutral surface. Product-specific
  model, permission, attachment, and action controls enter through slots rather
  than public Codex enums.
- Preserve common desktop text-entry behavior: Enter submits, Shift+Enter adds a
  line, and active IME composition never submits.
- Represent approval as a controlled, protocol-neutral request with explicit
  pending, approved, and rejected decisions; keep permission policies and
  transport-specific response payloads outside the component package.
- Keep command rendering independent from the shell transport: callers provide
  status and exit metadata, while stdout and stderr remain separately labeled
  presentational streams.
- Treat file changes as typed activities and accept structured diff lines from
  the caller; patch parsing, repository state, and apply/revert behavior stay
  outside the component package.
- Model reasoning as a dedicated disclosure: active reasoning initializes open
  with a `Thinking` label, while completed reasoning initializes collapsed with
  a `Thought` label. The state is initialized once rather than being forced to
  change when streaming completes.
- Model structured plans separately from proposed Markdown plans. A structured
  plan initializes expanded, uses pending, in-progress, and completed step
  states, and reports completed steps in its compact header. A proposed plan
  initializes collapsed while writing and expanded when completed.

## Markdown baseline

- Chat Markdown uses `16px` text with a `24px` line height and breaks long
  content anywhere rather than overflowing the thread.
- Paragraph and block spacing is approximately `11px`; headings use `20px`
  top and `10px` bottom spacing with a `600` weight.
- Heading sizes are `24px`, `20px`, `17px`, and `15px` across the six levels.
- List content starts approximately `21px` from the marker edge; adjacent and
  nested items use an `8px` gap.
- Blockquotes use a `4px` vertical rule, `24px` inline inset, and `8px`
  vertical padding.
- Inline code uses a `6px` radius and `1px Ă— 6px` padding.
- Code blocks use a compact language/action header, `8px` body padding,
  `12px` monospace text, a `20px` line height, and horizontal overflow unless
  wrapping is enabled.
- Tables use `14px` text, `24px` column separation, stronger header borders,
  horizontal overflow, and a `40rem` wide-block cap.
- Streaming content fades in over the relaxed transition duration and removes
  the animation under `prefers-reduced-motion`.

  `120ms` and when the block is within a `600px` vertical viewport margin.

## Reasoning and plan baseline

- Reasoning uses a full-width disclosure summary with a `6px` label/chevron
  gap, a `4px` body top inset, `12px` spacing between body blocks, and a chevron
  that becomes visible on hover or keyboard focus.
- Running reasoning is labeled `Thinking` and starts open. Completed reasoning
  is labeled `Thought` and starts collapsed.
- Structured plans start expanded. Their header uses `14px` summary text, a
  `16px` progress glyph, `6px` internal gaps, and `6px Ă— 8px Ă— 12px` header
  padding. The step viewport caps at `160px` with `8px` row spacing.
- Each structured step uses a `16px` status glyph, an explicit one-based index,
  `16px` chat text with a compact line height, and a line-through treatment for
  completed steps. The active step scrolls toward the viewport center.
- Proposed-plan cards use a `40px` header. Completed cards are titled `Plan`,
  expose download/copy/collapse actions, and start expanded. Streaming cards
  are titled `Writing plan` and start collapsed.
- These behaviors are represented through protocol-neutral public props; item
  mapping from app-server `reasoning`, `plan`, and `todo-list` events remains an
  adapter concern.

## Activity timeline baseline

- Related activity rows use a `4px` group gap; separate turn items use the
  standard `16px` activity gap.
- A collapsible turn summary uses `16px` secondary text, a `4px` label/chevron
  gap, a transparent `1px` border, a `6px` radius, subtle hover fill, and a
  focus-visible ring. Its chevron rotates by `90deg` when expanded.
- The divider sits `4px` below the summary and spans the full available width.
- Historical activity enters at `opacity: 0` and `translateY(-8px)`, reaching
  its resting position over `220ms` with `cubic-bezier(0.33, 1, 0.68, 1)`.
  Reduced motion keeps the opacity transition at `120ms` and removes movement.
- A host can render content before the toggle, persistent current work after
  the divider, and older collapsible activity. The persistent-content gap is
  conditional rather than inherent to every timeline.
- Active duration is `Working` for the first second and then `Working for
  {time}`. Completion is `Worked for {time}` and user interruption is `You
  stopped after {time}`.
- Duration formatting floors milliseconds to seconds, omits zero-value units,
  and uses compact day/hour/minute/second values such as `1m`, `1h 1m 1s`, or
  `1d 1s`. If duration data is unavailable, the collapsed fallback is `1
  previous message` or `{count} previous messages`.
- The public `ActivityTimeline`, `TurnDuration`, and controlled `AgentActivity`
  APIs reproduce these states without exposing turn protocol fields or product
  analytics.

The distributable implementation records these values as independently named
variables rather than copying the packaged selector graph. OpenAI Sans and
other bundled assets remain excluded; the token contract only declares the
font-family preference and a system fallback.

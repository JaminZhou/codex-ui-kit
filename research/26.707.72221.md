## Architecture observations

- The desktop shell is Electron and the primary Renderer is a React
  application built with Vite.
- Renderer styling combines generated utility CSS with semantic CSS custom
  properties for surfaces, text, borders, status, editor colors, and layout.
- The thread is a responsive content column backed by a virtualized turn list.
- A turn can contain user messages, assistant messages, command execution,
  patches, searches, MCP calls, approval review, subagent activity, and errors.
- Related work items are grouped more tightly than unrelated turns.
- Running, completed, failed, and pending states share one activity model rather
  than separate card systems.
- Activity rows can expose expandable details while keeping their summary on a
  compact single line.
- The composer is a separate bottom surface; floating and side panels are
  layered independently from the scrollable thread.

  pointer input, `aria-hidden`, and `tabIndex=-1`.

## Independent implementation decisions

- Use a protocol-neutral `AgentActivity` primitive for commands, patches,
  searches, subagents, and arbitrary tools.
- Use native `details` and `summary` semantics for the first expandable activity
  implementation.
- Keep all public class names and tokens under the `codex-ui-` namespace.
- Use system fonts; do not redistribute bundled OpenAI fonts.
- Reproduce measured semantic light and dark roles under the public
  `--codex-ui-` namespace without shipping extracted CSS.
- Keep virtualization and protocol adapters outside the first primitives
  milestone.
- Model the composer as a controlled, protocol-neutral surface. Product-specific
  model, permission, attachment, and action controls enter through slots rather
  than public Codex enums.
- Preserve common desktop text-entry behavior: Enter submits, Shift+Enter adds a
  line, and active IME composition never submits.
- Represent approval as a controlled, protocol-neutral request with explicit
  pending, approved, and rejected decisions; keep permission policies and
  transport-specific response payloads outside the component package.
- Keep command rendering independent from the shell transport: callers provide
  status and exit metadata, while stdout and stderr remain separately labeled
  presentational streams.
- Treat file changes as typed activities and accept structured diff lines from
  the caller; patch parsing, repository state, and apply/revert behavior stay
  outside the component package.
- Model reasoning as a dedicated disclosure: active reasoning initializes open
  with a `Thinking` label, while completed reasoning initializes collapsed with
  a `Thought` label. The state is initialized once rather than being forced to
  change when streaming completes.
- Model structured plans separately from proposed Markdown plans. A structured
  plan initializes expanded, uses pending, in-progress, and completed step
  states, and reports completed steps in its compact header. A proposed plan
  initializes collapsed while writing and expanded when completed.

- Structured plans start expanded. Their header uses `14px` summary text, a

- Each structured step uses a `16px` status glyph, an explicit one-based index,

  mapping from app-server `reasoning`, `plan`, and `todo-list` events remains an

## Activity timeline baseline

- Related activity rows use a `4px` group gap; separate turn items use the
  standard `16px` activity gap.
- A collapsible turn summary uses `16px` secondary text, a `4px` label/chevron
  gap, a transparent `1px` border, a `6px` radius, subtle hover fill, and a
  focus-visible ring. Its chevron rotates by `90deg` when expanded.
- The divider sits `4px` below the summary and spans the full available width.
- Historical activity enters at `opacity: 0` and `translateY(-8px)`, reaching
  its resting position over `220ms` with `cubic-bezier(0.33, 1, 0.68, 1)`.
  Reduced motion keeps the opacity transition at `120ms` and removes movement.
- A host can render content before the toggle, persistent current work after
  the divider, and older collapsible activity. The persistent-content gap is
  conditional rather than inherent to every timeline.
- Active duration is `Working` for the first second and then `Working for
  {time}`. Completion is `Worked for {time}` and user interruption is `You
  stopped after {time}`.
- Duration formatting floors milliseconds to seconds, omits zero-value units,
  and uses compact day/hour/minute/second values such as `1m`, `1h 1m 1s`, or
  `1d 1s`. If duration data is unavailable, the collapsed fallback is `1
  previous message` or `{count} previous messages`.
- The public `ActivityTimeline`, `TurnDuration`, and controlled `AgentActivity`
  APIs reproduce these states without exposing turn protocol fields or product
  analytics.

## Command execution baseline

- Generic commands initialize collapsed. Their compact summary is `Running
  command`, `Ran {command}`, or `Stopped {command}`; expanded completed and
  interrupted rows use the generic `Ran command` or `Stopped command` wording.
- Active commands append elapsed time after the first second. Command duration
  floors milliseconds and retains zero units after the minute boundary, such
  as `1m 0s`, `1h 0m 0s`, and `1d 0h 0m 1s`.
- A still-running process promoted out of the active turn uses `Started
  background terminal with {command}` and does not append the active timer.
- The embedded shell uses an `8px` radius, a heavier `1px` border, the code
  surface token, and clipped overflow. Its raw command has an explicit `$`
  prompt, `14px` monospace text, an initial two-line clamp, keyboard activation,
  and a copy action that appears on hover or focus.
- The output viewport uses medium `14px` monospace text, `8px` padding,
  preserved whitespace, horizontal scrolling, and a `140px` vertical cap. It
  follows appended output to the bottom and shows edge fades only where more
  content exists. Whitespace-only output renders `No output`.
- The compact footer uses `2px 10px 4px` padding. It stays visually empty while
  running, shows `Stopped` after interruption, a check plus `Success` for exit
  code zero, and `Exit code {code}` or `Exit code unknown` otherwise.
- A failed generic command retains the same collapsed `Ran {command}` wording
  as a successful command. Its failure signal is intentionally deferred to the
  expanded `Exit code {code}` footer; hosts can add compact metadata through the
  public `detail` slot when their product requires an always-visible signal.
- Shell expansion animates height and opacity with the shared basic transition;
  copy controls and command truncation remain keyboard accessible. Public props
  model these states without exposing process IDs, turn keys, or interrupt IPC.

## File change and inline diff baseline

- Patch items resolve to pending, streaming, applied, stopped, or rejected
  presentation states. Additions use `Creating`, `Created`, and `Stopped
  creating`; deletions use `Deleting`, `Deleted`, and `Stopped deleting`;
  edits use `Editing`, `Edited`, and `Stopped editing`; rejected operations use
  `Rejected`.
- File rows initialize collapsed. A collapsed applied row combines its action
  and path, such as `Edited src/status.ts`; an expanded applied row uses the
  generic `Edited file` summary and moves the current path into the inner diff
  header.
- Collapsed rows expose `+n` and `−n` statistics. Newly created files add a
  blue status dot and deleted files add a red status dot.
- The expanded diff shell starts `6px` below the summary, uses an `8px` radius
  and `1px` border, and clips the editor surface. Its compact header uses
  `14px` text, `10px` horizontal path inset, a subtle background, statistics,
  and a copy affordance.
- Inline diff content uses `12px` monospace text with a `20px` line height,
  separate old/new line-number columns, semantic addition and deletion
  surfaces, and horizontal scrolling unless wrapping is selected.
- The standard inline viewport caps at `240px`; a short streaming viewport caps
  at `100px`; fallback code caps at `160px`. Top and bottom fades appear only
  where more scroll content exists.
- Empty patch content is state-specific: deleted files report `Contents
  deleted`, renamed files report `File renamed without changes`, and an edit
  with no rendered hunks reports `No changes`. Other review surfaces can use
  `Binary file not shown`, `No content`, or their own host-provided fallback.
- Diff syntax rendering is a separate renderer path from Markdown code
  highlighting. The public primitive therefore accepts structured lines and
  host-rendered tokens without coupling the package to a parser, repository,
  protocol, or syntax-highlighting implementation.
- Height and opacity animate when completed content is disclosed. Streaming
  content is shown directly, and reduced-motion mode removes both disclosure
  and streaming-label animation.

## Approval and permission-request baseline

- Pending requests use a vertically stacked, overflow-clipped card with a
  `24px` radius, the input surface background, prominent desktop elevation,
  foreground text, and no desktop border. The card owns a container query so
  action groups reflow from rows to end-aligned columns in narrow hosts.
- The header uses `16px` horizontal/top padding and `12px` bottom padding. Its
  identity row combines a `16px` source icon and `14px` secondary label with an
  `8px` gap and `20px` line height. The title is `16px`, medium weight, and
  wraps anywhere; optional subtitle text uses the `14px` description role.
- A reason/details block sits below the title with a `2px` top inset and `4px`
  internal gap. `Reason` is represented as a semantic detail row rather than a
  generic code panel.
- The action form uses `8px` gaps and `8px 16px 16px` padding. Defaults are
  `Deny` and `Allow once`; an optional leading action defaults to `Always
  allow`. Scoped approval turns the primary action into a split button whose
  menu includes the one-shot option and `Allow this conversation` or a
  request-specific label such as `Allow similar commands` / `Allow all edits`.
- Enter approves and Escape denies while the pending surface is active.
  Approval hotkeys yield to dialogs and menus; Enter also yields to focused
  buttons and links. Loading disables actions and hotkeys.
- Command requests identify as `Terminal`, file patches as `Edit files`,
  managed network requests as `Internet access`, and filesystem requests as
  `Permissions`. Protocol response payloads and session-policy mapping remain
  host responsibilities.
- Command bodies use a `12px` monospace, medium-weight editor surface with an
  `8px` inset, `6px` radius, three-line collapsed clamp, `320px` expanded cap,
  and explicit `Expand` / `Collapse` actions. Patch bodies remain composable so
  the shared structured diff primitive can render the proposed file changes.
- The H5 and Electron matrices cover command, patch, network, filesystem,
  connector/loading, scoped menu, explicit outcome, keyboard, theme, and
  narrow-window states without importing private product code or assets.

## Search, MCP, and connector-call baseline

- Code-search activities use `Searching for {query} in {path}` while active and
  `Searched for {query} in {path}` after completion. A missing path removes the
  final phrase, while a missing query falls back to `Searching for files` or
  `Searched for files`.
- A standalone web-search row uses `Searching the web for {query}` or `Searched
  the web for {query}`. Grouped web searches initialize collapsed, show the
  most recent unfinished detail while active, and collapse to `Searched the
  web` without a query suffix once every entry completes.
- The grouped web header uses a `16px` secondary globe icon. Expanded entries
  use `16px` sans text with a `24px` line height, optional `14px` favicons, and
  a `320px` vertical cap. Height and opacity use the shared basic transition.
- MCP, connector, browser, and desktop-tool calls share one activity-card
  shape. Their label is provider/tool-specific, uses a `16px` source or tool
  icon, and animates while the call is active. Normal completed calls initialize
  collapsed; a running call without a result does not expose a disclosure.
- Generic fallback labels derive from the tool name after removing a known
  connector prefix. Known integrations can provide precise active/completed
  verbs such as `Searching issues` / `Searched issues`; browser fallbacks use
  `Used Chrome` or `Used the browser`.
- Expanded tool results accept text and other rendered blocks, a danger result,
  structured JSON, or the exact empty state `Tool returned no content`. Raw
  text, error, and structured-code viewports cap at `192px`.
- A raw-output icon appears on hover or keyboard focus and opens a separate
  inspection surface owned by the host. The public component exposes the event
  and payload without taking a dependency on a modal system or transport.
- Rich MCP App widgets, embedded resources, and media blocks build on this
  call shell but remain part of the resource/artifact surface; no private MCP
  widget runtime or connector metadata is copied into the package.

## Error, warning, notice, retry, and interruption baseline

- The reusable status banner uses a `16px` radius, a main-surface background,
  compact `8px 8px 8px 12px` insets, `14px` text, a subtle shadow, and a
  semantic border in browser surfaces. Electron replaces the full border with
  a `0.5px` heavy ring so the shell stays crisp on native desktop displays.
- Neutral, information-accent, warning, and error banners share the same
  geometry. Warning uses the editor-warning foreground and a low-opacity
  warning surface; error uses the error foreground and validation-error
  surface; information accent uses the link foreground and input background.
- Horizontal, vertical, and leading-icon layouts are observed. Action groups
  use an `8px` gap and support outline, ghost, primary, and danger actions.
  Horizontal banners can reflow their actions to a centered wrapped row at the
  `400px` narrow boundary. Live-region role and politeness are selected by the
  caller rather than inferred from color alone.
- Generic system errors use the leading-icon vertical layout and break arbitrary
  content anywhere. Usage-limit errors instead use an inline transcript notice
  between two flexible border rules; its content is secondary text with an
  optional icon, trailing guidance, and a wrapping mode for long messages.
- Stream errors use `16px` chat text with an 80%-secondary foreground.
  Reconnect attempts use tabular `{attempt}/{maximum}` progress. HTTP 429 and
  server-overload states change the label to `Server is busy, reconnecting`;
  other disconnects use the caller-provided message or `Reconnecting`.
- Additional stream details disclose beneath the summary with a `4px` top gap,
  height/opacity motion, and a chevron that appears on hover, focus, or while
  expanded. A terminal failure can expose an explicit retry action; transport
  request IDs and raw protocol errors remain host-provided content.
- Auto-review interruption uses the inline notice shape with `Turn ended by
  Auto-review` and optional guidance explaining that repeated denials ended the
  turn. User interruption continues to use the turn divider language `You
  stopped after {time}`.
- The public `StatusBanner`, `InlineNotice`, and `StreamNotice` APIs model these
  surfaces without exposing private error enums. H5 acceptance covers light,
  dark, and `420px` no-overflow rendering. Electron 42 acceptance covers
  standard dark and `820 × 680` compact light rendering, system-font fallback,
  fixed chrome/composer, and Renderer-owned scrolling.

## Composer shell and autosizing baseline

- The composer switches between a compact single-line shell and a multiline
  surface. Attachments, explicit line breaks, or text that no longer fits the
  compact input select multiline; otherwise the shell returns to single-line.
  Hosts can request either layout when deterministic placement is required,
  but attachments and explicit line breaks retain structural priority.
- Single-line layout is a three-column footer: leading actions, a
  `minmax(0, 1fr)` input, and trailing controls. It uses a `22px` radius,
  `8px` column gaps, `8px` horizontal plus `4px` vertical shell padding, and a
  `36px` input row whose text line is `20px` high.
- Multiline layout uses a `20px` radius. Its editor has a `44px` minimum height,
  a `25dvh` scroll cap, `12px` horizontal inset, and a `4px` gap before the
  footer. The footer uses `8px` horizontal/bottom inset, `5px` between its main
  regions, and `4px` between inline controls.
- Composer controls and the send/stop action are `28px` high. The primary
  action is circular, uses the foreground surface, and reaches 50% opacity when
  disabled or submitting. It does not translate on hover.
- The shell has no full-pixel border. Its browser elevation combines a `0.5px`
  heavy ring with low-opacity `3px / 7.5px` and `20px` shadows. Electron keeps
  the ring but reduces the ambient spread to `16px`. The input surface is
  translucent with backdrop blur; dark Electron rendering increases its
  opacity. Forced-colors mode replaces the shadow with a one-pixel outline.
- Clicking non-interactive shell chrome focuses the editor. Links, buttons,
  inputs, selects, editable/draggable content, roles, and explicit tab stops
  retain their own interaction. Enter submits, Shift+Enter inserts a line, and
  active IME composition never submits.
- Keyboard focus preserves the platform-specific composer elevation and adds a
  two-pixel focus ring. Forced-colors mode replaces it with the system Highlight
  outline so the textarea never relies on the caret as its only focus signal.
- The shell keeps overflow visible so protocol-neutral slots can own
  non-portaled menus, tooltips, and mention pickers without clipping. Individual
  scrollable editor and attachment regions own their own overflow boundaries.
- The public implementation uses a native controlled textarea rather than the
  sampled product's rich-text editor runtime. Its independent width probe,
  ResizeObserver path, height synchronization, DOM geometry, keyboard contract,
  and accessibility states reproduce the observable shell behavior without
  importing private editor, protocol, analytics, or IPC code.
- H5 acceptance covers interactive attachment-driven multiline, compact,
  autosized multiline, running/narrow, and disabled states. Electron 42 covers
  the same matrix under system theme/font, standard and compact windows,
  Renderer scrolling, resizing, and focus.

## Composer context and queue baseline

- Composer attachments form a one-line horizontal tray with hidden scrollbar,
  `8px` inset, `8px` item gaps, and independent overflow. The observed state
  family includes image previews, file/folder cards, pasted text, selections,
  context references, upload progress, failure, opening, and removal. The
  public `ComposerAttachment` models that visual contract through protocol-
  neutral `pill`, `card`, and `image` layouts plus ready, uploading, and error
  states.
- Mention suggestions attach above and overlap the composer shell by `14px`.
  The tray extends `16px` past each composer edge, caps at `177px`, groups
  results such as files, chats, apps, skills, agents, and servers, and owns its
  vertical scrolling. Loading, empty, active, disabled, pointer, Arrow,
  Home/End, Enter, and Escape behavior remain independent of the sampled rich-
  text editor runtime.
- Plan, Goal, Review, and host-defined modes use the same `28px` footer control
  height and tertiary foreground as adjacent composer controls. Clearable modes
  exchange their leading glyph for an × affordance on hover or keyboard focus.
- Queued prompts occupy a hidden-scrollbar list capped at `30dvh`. Rows are
  `32px` high with a hover/focus drag handle, one-line summary, Steer or Retry,
  delete, and an actions menu. Paused, editing/dragging, and interrupted-queue
  states expose warning, opacity, Resume, edit, queue-toggle, pointer reorder,
  and Alt+Arrow keyboard reorder behavior.
- `ComposerMentionMenu`, `ComposerModeIndicator`, and `QueuedPromptList` expose
  only caller-owned IDs, labels, content, and callbacks. No renderer editor
  nodes, protocol records, analytics identifiers, or private state types enter
  the package. H5 and Electron matrices exercise the same public components.

## Subagent and delegated-work baseline

- Delegated work renders at three distinct levels: compact activity inside the
  transcript, a collapsible `Subagents` section in the thread summary, and a
  dedicated side panel that can navigate into a selected child transcript.
- A standalone transcript event uses the agent avatar plus `started working`,
  `updated`, or `interrupted` copy. The row is a keyboard-focusable button when
  the host can open that child thread.
- Consecutive multi-agent events collapse into at most three inline chips. Each
  chip is `28px` high, caps at `192px`, uses a `16px` deterministic avatar, a
  full radius, `6px` internal gap, and a light border. Additional agents use
  `and {count} other subagent(s)` before the shared status.
- Group status priority is `interrupted`, then `updated`, then `finished` when
  every agent is done, otherwise `started working`. New chips enter over
  `280ms` from `translate(-8px, 4px) scale(.92)` and remove the animation under
  reduced motion.
- The summary section groups up to four inline agent avatars and reports
  `{count} working` plus a secondary `{count} done`. Non-inline agents remain
  individual rows with the active `is working` shimmer, optional role/model
  tooltip, and addition/deletion statistics. A completed row-only section
  initializes collapsed.
- The dedicated panel uses `12px × 20px` outer padding. `Active` shows up to
  four items with two-line previews and the exact `No active subagents` empty
  state; `Done · {count}` shows up to ten items with one-line previews. Rows use
  `24px` avatars, a `4px` gap between rows, `14px` text, and `20px` preview line
  height. Missing active copy falls back to `Working`; waiting copy falls back
  to `Thinking`; completed rows without copy omit the preview.
- A selected child transcript uses a `48px` header with a `24px` back button,
  `24px` avatar, `8px` gap, `16px` horizontal inset, and a heavy bottom border.
  Transcript hydration, nested-thread validation, routing, and transport state
  remain host responsibilities.
- The public API names independent `SubagentItem` and
  `SubagentActivityItem` models. It does not expose app-server thread records,
  background-agent atoms, route stores, private IPC, or bundled avatar assets.

The distributable implementation records these values as independently named
variables rather than copying the packaged selector graph. OpenAI Sans and
other bundled assets remain excluded; the token contract only declares the
font-family preference and a system fallback.

## Interactive control and overlay baseline

- Base buttons share a border-aware inline-flex shell, a `4px` content gap,
  compact medium-weight text, and primary, secondary, outline, ghost, and
  danger roles. The observed toolbar control is `28px` square with a `20px`
  icon; disabled controls retain their geometry at `40%` opacity. The public
  API also exposes `20px` small and `36px` large geometry for adjacent sampled
  control families.
- Icon buttons expose an accessible label and optional pressed state without
  relying on a native `title`. The tooltip layer supplies the visible label.
  Tooltips open immediately for keyboard focus and after a `700ms` pointer
  delay, default above the anchor with a `2px` offset, and close on blur or
  Escape. Their measured shell uses an `8px` radius, `4px × 8px` inset, compact
  text, optional shortcut, and a `320px` maximum width.
- Menus and popovers portal to the current Renderer document, flip when the
  requested side has insufficient space, and clamp to an `8px` viewport edge.
  The surface uses a translucent elevated background, `0.5px` ring, `12px`
  radius, `4px` outer inset, desktop blur, and an `8px / 16px` shadow. Standard
  and flyout menus have a `180px` minimum width and a `4px` anchor offset.
- Menu rows use `8px` horizontal and Electron-observed `5px` vertical insets,
  an `8px` radius, secondary icons and shortcuts, a visible hover/focus fill,
  and `50%` disabled opacity. Public rows cover labels, descriptions,
  separators, destructive actions, checkboxes, and portalled submenus.
- Menu and listbox triggers open with click or Arrow keys. Arrow Up/Down,
  Home/End, Escape, Tab, native activation, focus restoration, outside pointer
  dismissal, and disabled-item skipping are covered without importing a
  private focus runtime. A shared overlay owner keeps a parent menu open while
  its portalled submenu receives pointer events.
- Selects reuse the listbox surface with a trigger-width floor of `180px`, a
  `220px` option viewport, selected checkmark, optional icon and description,
  disabled options, and caller-owned values. Wider menu surfaces can request
  the observed `288px` role.
- H5 acceptance verifies a `180px` root menu and submenu inside a `1280 × 720`
  viewport, shared `12px` radii and `4px` insets, outside-close behavior, and
  controlled selection. Electron 42 acceptance records the same control
  heights, shared overlay ownership, light/dark theme inheritance, desktop
  portal geometry, and viewport containment through the self-capture harness.

## Resource, source, artifact, and generated-image baseline

- End-of-turn resources share one divided list and expose the first three rows
  before `Show {count} more`. Observed row families include local files,
  websites, Google Docs/Sheets/Slides/Drive, generated apps, and images. Their
  transport records remain outside the public component.
- A resource row uses a `40px` square visual well with an `8px` radius and a
  `24px` icon or thumbnail. The row is approximately `60px` high, keeps title
  and secondary metadata on single lines, and exchanges its normal subtitle
  for an open target/action on hover or keyboard focus. Trailing actions remain
  independently operable above the full-row open target.
- File subtitles use the user-visible family and extension, such as
  `Document · MD`, `Spreadsheet · XLSX`, `Slides · PPTX`, or `Image · PNG`.
  Websites use `Website`; Drive items use `Google Docs`, `Google Sheets`,
  `Google Slides`, or `Drive`. Opening, downloading, sharing, drag payloads,
  filesystem paths, and connector navigation are host callbacks.
- Source summaries are titled `Sources`, expose a compact subset plus
  `View all`, and use `18px` previews with a `3px` radius. File detail uses the
  visible phrase `Attached to the conversation`; web results use `Web search`.
  The measured summary panel width is `300px`.
- Artifact families include external resources, websites, Drive content,
  generated apps, files, and generated images. The empty state is
  `No artifacts yet`. The public `ArtifactList` composes resource primitives
  rather than accepting private artifact unions.
- A single generated image preserves its natural ratio up to `400px`. Multiple
  images use an `8px` gap and natural widths when they fit; an overflowing set
  exposes at most four square slots at once. The overflow badge and previous/
  next controls are `24px` dark translucent pills positioned `8px` from the
  lower-right edge.
- Generated-image buttons and pending placeholders use a `16px` radius.
  Square slots crop with `object-fit: cover`; natural slots contain the image.
  Pending generation reserves at least four total slots and pulses unless the
  user requests reduced motion. Images use a no-referrer policy, adopt their
  intrinsic ratio after load when dimensions are absent, and retry two failed
  loads before entering a terminal unavailable state.
- The preview is a portalled modal with previous/next navigation, an explicit
  close action, `Download`, Escape and Arrow keys, a focus loop, focus return,
  backdrop dismissal, and background-scroll locking. The public API accepts
  only caller-owned image IDs and URLs.
- H5 acceptance records `60px` resource rows, a `300px` source list, zero page
  overflow, three-to-five reveal, image paging, and modal keyboard behavior.
  Electron 42 self-capture verifies `1180 × 820` dark and `820 × 680` light
  windows, `16px` gallery radii, `60px` rows, `300px` source geometry, pending
  placeholders, portalled preview focus, font/theme inheritance, and no
  horizontal overflow. Captures remain under `/private/tmp`.

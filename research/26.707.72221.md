## Architecture observations

- The desktop shell is Electron and the primary Renderer is a React
  application built with Vite.
- Renderer styling combines generated utility CSS with semantic CSS custom
  properties for surfaces, text, borders, status, editor colors, and layout.
- The thread is a responsive content column backed by a virtualized turn list.
- A turn can contain user messages, assistant messages, command execution,
  patches, searches, MCP calls, approval review, subagent activity, and errors.
- Related work items are grouped more tightly than unrelated turns.
- Running, completed, failed, and pending states share one activity model rather
  than separate card systems.
- Activity rows can expose expandable details while keeping their summary on a
  compact single line.
- The composer is a separate bottom surface; floating and side panels are
  layered independently from the scrollable thread.

## Representative measurements

These measurements describe relationships, not a promise of pixel parity:

- Default activity separation: approximately `16px`.
- Grouped activity separation: approximately `4px`.
- User-message bubble width cap: approximately `77%` on desktop.
- User-message bubble radius: approximately `16px`.
- User-message bubble inset: approximately `8px` vertical and `12px`
  horizontal.
- Common radius steps cluster around `6px`, `8px`, `10px`, and `12px`.
- Basic state transitions are around `150ms`; relaxed transitions around
  `300ms`.

## Independent implementation decisions

- Use a protocol-neutral `AgentActivity` primitive for commands, patches,
  searches, subagents, and arbitrary tools.
- Use native `details` and `summary` semantics for the first expandable activity
  implementation.
- Keep all public class names and tokens under the `codex-ui-` namespace.
- Use system fonts; do not redistribute bundled OpenAI fonts.
- Provide original light and dark palettes with semantic roles instead of
  copying the shipped token table.
- Keep virtualization and protocol adapters outside the first primitives
  milestone.
- Model the composer as a controlled, protocol-neutral surface. Product-specific
  model, permission, attachment, and action controls enter through slots rather
  than public Codex enums.
- Preserve common desktop text-entry behavior: Enter submits, Shift+Enter adds a
  line, and active IME composition never submits.
- Represent approval as a controlled, protocol-neutral request with explicit
  pending, approved, and rejected decisions; keep permission policies and
  transport-specific response payloads outside the component package.

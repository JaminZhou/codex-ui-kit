## Scope correction

The `26.715.31925` conclusion treated the absence of a new known conversation
anchor as evidence that no new component category was required. That method
could only rediscover categories already in the checklist. It did not establish
a complete denominator for conversation states, workspace panels, application
navigation, settings, integrations, or feature-gated routes.

The current baseline therefore invalidates the previous product-level
`Complete` claim. Existing components remain useful independent primitives,
but their coverage is `partial` until current-build runtime states and
compositions are observed and verified.

## Runtime boundary

The available Computer Use runtime explicitly blocks access to
`com.openai.codex` for safety reasons. No Computer Use bypass was attempted.

With explicit user authorization, a second application process was launched
with a temporary Chromium user-data directory and a CDP endpoint bound only to
`127.0.0.1`. CDP connected successfully to the `app://-/index.html` Renderer.
The temporary Chromium profile did not isolate Codex's application-level
account, project, or navigation state, so it must not be treated as a clean
profile. Existing conversations were not used for research. One accidental
navigation to an existing view was stopped immediately, and the prepared test
prompt was confirmed absent and unsent.

The allowed runtime sample used a manually opened blank chat in
`codex-ui-kit`, changed permission mode from Full access to Ask for approval,
and submitted a read-only Git status request. It observed:

- the global sidebar and top navigation controls;
- the Pull requests index with filter tabs, search, and split empty/detail
  layout;
- the Sites index with search, refresh, create, sharing, and overflow actions;
- the Scheduled tasks index with status filters, search, suggestions, and task
  actions;
- the Plugins and Skills indexes with search, category/filter, refresh,
  management, and creation actions;
- the blank new-chat state and portalled project selector;
- Composer project, local run location, branch, permission, model, attachment,
  dictation, and submit/stop states;
- the run-location choices for local, web/cloud, and worktree execution; the
  current-branch menu; and the model, effort, and speed menu;
- the permission menu's Ask for approval, Approve for me, Full access, and
  custom configuration choices;
- a basic user turn, `Thinking`, `Worked for 8s`, assistant completion, and an
  expandable `Ran a command` activity;
- grouped command activities with per-command disclosure, a no-output failure
  with exit code 7, a successful 220-line output, and a 144-pixel-tall
  scroll/fade output container;
- a queued follow-up card with delete and actions controls, sequential
  execution after the active turn, and the running Composer `Steer` control;
- the `Stop` control, `You stopped after …` summary, stopped command state, and
  Background processes section;
- Markdown heading, unordered and ordered lists, table, inline code, code
  block, blockquote, and external link rendering;
- the right-side Review tab, file-diff list, panel expansion controls, branch
  summary, and commit/pull-request actions;
- the Sources summary, `View all` transition, 667-pixel right-side Sources
  tabpanel, and the bottom terminal tabpanel with a named tab and accessible
  terminal input;
- the full-page Settings shell with search and grouped navigation, plus
  read-only samples of General, Appearance, Plugins, Browser, Computer use,
  Hooks, Connections, Git, Environments, and Worktrees.

The CDP Renderer viewport was also probed at 2560×1410, 1180×820, and 820×680.
The left navigation remained 274 pixels wide. The Composer content width was
712 pixels at the first two sizes and shrank to 490 pixels at 820 pixels. The
right Environment panel remained laid out beyond the simulated narrow
viewport, which exposes a P0 responsive-shell risk: narrow mode needs an
explicit sidebar/panel policy rather than relying on content shrink alone.
Electron did not map this application target to a Browser window ID, so these
are Renderer viewport observations, not proof of real native-window resize
behavior.

The permission mode remained Ask for approval. A read-only access probe outside
the project did not require approval, so no permission-request card was
observed and that surface remains `not_sampled`.

The test changed no repository file or global Codex setting. The disposable
test chat remained in Ask for approval mode. It created and closed one bottom
terminal tab but ran no command in that terminal. This is one dark-theme sample
with several read-only routes, not a complete runtime traversal. Every
unobserved state, theme, overlay, and native/Electron surface remains
unverified.
